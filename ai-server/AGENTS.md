# AI Server Review Guide

## Review guidelines

- 리뷰 코멘트는 한국어로 작성한다.
- 실제 버그, 데이터 유실 가능성, RAG 답변 품질 저하, 성능 문제, 운영 장애 위험을 우선 지적한다.
- 단순 스타일 취향이나 nit는 실제 결함 가능성이 있을 때만 지적한다.
- FastAPI endpoint의 요청/응답 계약이 Spring Boot `FastApiClient`와 일치하는지 확인한다.
- 문서 업로드 실패 시 ChromaDB 저장 상태와 Spring Boot 문서 상태가 어긋날 가능성이 있는지 확인한다.
- PDF, DOCX, PPTX, XLSX 파서 분기가 확장자와 예외 처리 기준을 일관되게 따르는지 확인한다.
- 청킹 로직이 한글 문서의 제목, 표, 목록, 페이지 경계를 과도하게 깨뜨리지 않는지 확인한다.
- ChromaDB metadata에는 문서 ID, 원본 파일명, 헤더 경로처럼 출처 표시와 삭제에 필요한 값이 유지되는지 확인한다.
- Ollama 모델명, 임베딩 모델명, Chroma host 같은 운영 설정은 환경변수로 분리되는지 확인한다.
- Langfuse 추적은 키가 없을 때 비활성화되어도 핵심 질의응답 흐름을 깨지 않는지 확인한다.

## Clean code checklist

- 파싱, 정규화, 청킹, 임베딩, 저장, 질의응답 단계를 함수 단위로 분리했는지 확인한다.
- 외부 라이브러리 호출 실패는 사용자에게 이해 가능한 오류로 변환되는지 확인한다.
- 파일 확장자, chunk size, overlap, topK 같은 정책값은 상수나 환경변수로 관리되는지 확인한다.
- 큰 문서 처리에서 불필요한 전체 복사나 과도한 메모리 사용이 없는지 확인한다.
- 동기/비동기 함수가 섞일 때 blocking 호출이 이벤트 루프를 과도하게 막지 않는지 확인한다.

## Project-specific focus

- Two-Pass 청킹은 Markdown header 경계를 우선 보존하고, 큰 청크만 재분할하는 구조를 유지한다.
- 한글 문서에서 3개 이상 개행 정규화와 50자 수동 overlap이 검색 품질을 해치지 않는지 확인한다.
- 문서 삭제 또는 재업로드 기능이 추가될 때 ChromaDB collection의 기존 chunk 정리 여부를 반드시 확인한다.
- `/query/stream`은 토큰 스트리밍 중 오류가 나도 클라이언트가 종료 이벤트나 오류 이벤트를 받을 수 있어야 한다.
